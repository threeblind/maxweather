# 注目チーム登録 詳細設計

- 作成日: 2026-06-20
- 対象: 第16回大会フロントエンド
- 対象ファイル: `index_16.html`, `app_16.js`
- 目的: ユーザーが好きな大学を 1〜3 校登録し、順位表・地図・グラフで強調表示し、順位変動時のみ通知する
- 前提: サーバー変更なし。クライアント側の `localStorage` のみで完結する

## 1. 仕様の要点

注目チーム登録は、観戦中の大学を継続追跡するための補助機能とする。

- 登録数は 1〜3 校
- 登録 UI はメニューに追加しない
- 登録はエントリーリストから行う
- 表示は順位表、地図、グラフの3箇所で強調する
- 順位変動通知は注目チームのみ対象とする
- 通知は既存の定期通知基盤を流用する
- 注目チームの通知は 1 回のチェックにつき 1 件にまとめる

## 2. 体験設計

### 2.1 ユーザーの操作

1. エントリーリストで大学を見つける
2. 行の右端にある星ボタンを押して注目登録する
3. 登録中は星が塗りつぶされ、注目状態が分かる
4. もう一度押すと解除する
5. 3 校登録済みなら、それ以上は追加できない

### 2.2 注目状態の見え方

- 順位表: 該当チームの行を強調
- 地図: 該当チームのマーカーを強調
- グラフ: 該当チームの線を強調し、他チームを弱める
- 画面内のどこかに「注目チーム 2/3」のような残数表示を出す

## 3. UI 要件

### 3.1 登録 UI の設置場所

エントリーリストに置く。新しいメニュー項目は増やさない。

採用案:

- 各大学行の右端に星ボタンを置く
- PC では `☆ 注目` / `★ 注目中` の切り替え
- SP でも同じ機能を維持し、タップ領域を十分に確保する

### 3.2 ボタン状態

- 未登録: `☆ 注目`
- 登録中: `★ 注目中`
- 上限到達時に未登録のボタンを押した場合は登録できない
- 上限到達時は、ボタンの見た目を無効状態にするか、押下時に軽い案内を出す

### 3.3 表示文言

用語は `注目チーム` に統一する。

表示例:

- `注目チーム 1/3`
- `注目中の大学はありません`
- `3校まで登録できます`

### 3.4 説明文

エントリーリスト上部に、注目登録の使い方を1行だけ表示する。

表示例:

- `☆を押して注目チームに登録できます。最大3校まで。`
- 必要なら補足として `順位変動時に通知します` を小さく添える

説明は長くしない。初見で操作が分かる最小限の文だけにする。

## 4. 注目チームの動作要件

### 4.1 登録・解除

- クリックで追加/解除を切り替える
- 登録対象は `team.id`
- 表示名ではなく ID で保持する
- 登録順は保存してもよいが、表示順に意味は持たせない

### 4.2 上限

- 最大 3 校
- 4 校目以降は登録不可
- 既登録の解除で空きが戻る

### 4.3 永続化

- ブラウザ再読込後も注目状態を維持する
- 端末ごと、ブラウザごとに独立した状態とする

## 5. 通知要件

### 5.1 基本方針

- 通知は既存の定期通知基盤を流用する
- 注目チームの順位変動を、定期チェック時にまとめて判定する
- 通知本文は定期速報とは別テンプレートを使う
- 初回表示では通知しない

### 5.2 通知対象

注目チームについて、次の変動を対象にする。

- 順位上昇
- 順位下降
- 首位交代に関与した変動

### 5.3 通知のまとめ方

- 1回のチェックで複数の注目チームが変動しても通知は 1 件
- 通知文面の中で複数大学の変動を列挙する前提でよい
- 速報通知とは別系統なので、同じ時刻に両方届く可能性はある

### 5.4 重複防止

- 同じ変動を再通知しない
- `localStorage` に最終判定状態を保存する
- ページ再読み込み後も前回通知の重複を避ける

## 6. localStorage 設計

保存は最小限にする。

### 6.1 保存キー

- `favoriteTeams`
- `favoriteTeamsNotificationState`

### 6.2 保存内容

```json
{
  "favoriteTeams": [3, 7, 12],
  "favoriteTeamsNotificationState": {
    "lastCheckedAt": "2026-06-20T10:05:00+09:00",
    "lastNotifiedEventKey": "2026-06-20T10:00:00+09:00:favorites-change"
  }
}
```

### 6.3 各項目の意味

- `favoriteTeams`
  - 注目中の `team.id` 配列
  - 1〜3 件

- `favoriteTeamsNotificationState`
  - 最後に確認した時刻、または最後に通知したイベントを記録する
  - 同じチェック結果を二重通知しないために使う

### 6.4 保存しないもの

- チーム名
- カラー
- 現在順位
- 通知文面
- UI の一時状態

これらは毎回データソースから復元する。

## 7. 画面内の強調ルール

### 7.1 順位表

- 注目チームの行をハイライトする
- 既存の順位表の配色や強調ルールと衝突しないようにする
- 追跡中の表示と見分けられるよう、背景色または左ボーダーを使う

### 7.2 地図

- 注目チームのマーカーを強調する
- 可能ならサイズ、z-index、ポップアップ、または輪郭で差を付ける
- 他のマーカーより見つけやすくする

### 7.3 グラフ

- 注目チームの線を太くする、または不透明度を上げる
- 非注目チームは薄くする
- ひと目で追跡対象が分かることを優先する

## 8. 実装方針

### 8.1 データ構造

`app_16.js` に注目チーム用の状態を追加する。

必要な状態:

- 注目チームの `Set` または `Array`
- 注目通知の最終判定情報
- 注目状態の読み込み・保存ヘルパー

### 8.2 エントリーリスト連携

- エントリーリスト描画時に星ボタンを付ける
- クリックで `favoriteTeams` を更新する
- 更新後は順位表、地図、グラフの再描画または再強調を行う

### 8.3 通知連携

- 既存の定期通知処理の中で注目チームを判定する
- 定期速報処理とは分岐を分ける
- 注目チーム通知はまとめて 1 件にする

### 8.4 更新タイミング

次のタイミングで状態を反映する。

- 初期読み込み時
- `fetchEkidenData()` 後
- 注目チームの登録/解除時
- 定期通知チェック時

## 9. エッジケース

- 注目チームが 0 校のときは通知判定を行わない
- 3 校登録済みで新規追加しようとした場合は登録不可
- 監視対象が現行データに存在しない場合は無視する
- データ欠損時は通知を出さず、次回更新を待つ
- ブラウザが通知未対応の場合は、画面内の強調だけにする

## 10. 実装順

1. `localStorage` の読み書きヘルパーを追加する
2. エントリーリストの各行に注目ボタンを追加する
3. 注目状態に応じた UI 強調を入れる
4. 既存通知基盤に注目チーム判定を差し込む
5. 重複通知防止を入れる
6. 画面再描画時の状態復元を確認する

## 11. 受け入れ条件

- エントリーリストから 1〜3 校の注目登録ができる
- 登録状態が再読込後も維持される
- 順位表、地図、グラフで注目チームが分かる
- 注目チームの順位変動時のみ通知が出る
- 3 校同時変動でも通知は 1 件にまとまる
- サーバー側の変更は不要

## 12. 付記

この機能は「速報を読む」ためではなく、「特定大学を追い続ける」ための機能として扱う。
そのため、文言は短く、操作は少なく、表示は一貫させる。
